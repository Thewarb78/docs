Proposed Namespace Model

jobs namespace

Contains:

- General Python jobs
- Spark submitter/driver pods (client mode)
- Lightweight orchestration/runtime logic

spark namespace

Contains:

- Spark executor pods only
- Heavy distributed compute workloads
- Platform-managed quotas/policies/security controls

Rationale

This separates:

- lightweight application/orchestration workloads
  from
- heavy Spark compute workloads

Benefits:

- Central control of Spark executor capacity
- Easier quota/governance for distributed compute
- Cleaner separation of platform-managed compute resources
- General jobs namespace not impacted by Spark executor scaling

RBAC Requirements

Spark submitter pods running in "jobs" namespace must be allowed to create/manage resources in "spark" namespace.

Required approach

1. Create ServiceAccount in "jobs" namespace

Example:

- "jobs:spark-submit"

2. Create Role in "spark" namespace

Permissions required for Spark executors/resources, e.g.:

- pods
- services
- configmaps
- pvc (if used)

Typical verbs:

- create
- get
- list
- watch
- patch
- delete

3. Create RoleBinding in "spark" namespace

Bind:

- "jobs:spark-submit"

to:

- Spark executor management Role in "spark"

Additional Considerations

- Secrets required by executors (e.g. Hadoop/core-site config, image pull secrets) must exist in "spark" namespace
- Logs/events/debugging will span "jobs" and "spark" namespaces
- Spark submitter uses:
  - "spark.kubernetes.namespace=spark"
- Submitter pod itself still runs in "jobs"
